# Engineering Autonomy & Strategic Planning Specification
**Document Version**: 1.0.0
**Date**: 2026-02-28
**Prepared by**: Tala (Autonomous AI Assistant)

## 1. Overview
The Engineering Autonomy & Strategic Planning (EASP) layer is a significant architectural enhancement to the Tala AgentService. It transitions the agent from a reactive tool-user to a proactive, goal-oriented software engineer. This is achieved through reinforced system prompts, integrated goal management, and self-aware reflection logging.

## 2. Core Protocols

### 2.1 Engineering Autonomy Protocol (EAP)
The EAP mandates direct workspace modification. The agent is strictly prohibited from creating "standalone" scripts (e.g., `feature_test.js`) unless explicitly requested. Instead, it must:
1.  **Analyze**: Use `list_files` and `read_file` to understand the existing implementation.
2.  **Integrate**: Use `patch_file` or `write_file` to modify the production codebase directly.
3.  **Validate**: Use `system_diagnose` to verify linting and build status after modifications.

**Prompt Snippet (Internal):**
> "You are a fully autonomous software engineer with 'File Control'. If a change is requested, execute the tool calls immediately. 'render_a2ui' is ONLY for visual dashboards, NOT a substitute for real code changes."

### 2.2 Strategic Planning & Reflection Protocol (SPRP)
The SPRP ensures multi-turn task consistency and self-correction.
1.  **Goal Synchronization**: The `task_plan` and `manage_goals` tools are now synchronized with the `GoalManager` backend. Planning actions are persisted in the agent's context as a `[SHIP'S LOG: GOAL GRAPH]`.
2.  **Stability Awareness**: The agent's context now includes a `[REFLECTION LOGS: SYSTEM STABILITY]` section, summarizing recent errors, latency, and tool failures observed by the `ReflectionEngine`.

## 3. Tool Enhancements

### 3.1 `task_plan` (Unified Planning)
*   **Old Behavior**: Rendered a visual tree to the user but did not affect the agent's internal state.
*   **New Behavior**: Automatically injects sub-goals into the `GoalManager`. These goals reappear in the system prompt for all subsequent turns, ensuring the agent "remembers" its roadmap even after a reload or context shift.

### 3.2 `getReflectionSummary` (Self-Awareness)
This internal method aggregates artifacts from `userData/memory/reflections` and `userData/memory/proposals`. It exposes:
*   **Recent Observations**: Evidence collected by the system (e.g., a 50% increase in Ollama latency).
*   **Pending Proposals**: Changes generated by the `ProposalEngine` that are awaiting user approval in the Reflection Panel.

## 4. Stability Improvements

### 4.1 Greedy Tool Call Parsing
To resolve 400 errors from local inference engines (Ollama/Llama.cpp), the `AgentService` now uses a "Greedy Brace Extraction" strategy. It robustly finds the outermost valid JSON block, ignoring trailing hallucinations or malformed XML tags.

### 4.2 Interruptible Retry Logic
The `streamWithRetry` mechanism is now integrated with the `AbortController`. Pressing the **STOP** button in the UI immediately cancels any active exponential backoff sleep timer, providing instant response to user cancellation requests.
