## 1. Overview
This document defines the schema and mutation rules for the Tala configuration system. All persistent settings are stored in `app_settings.json` and managed by the `SettingsManager` service.

## 2. Storage Location
-   **Path**: `%USERDATA%/app_settings.json`
-   **Format**: JSON

## 3. Schema Definition (Simplified)

### 3.1. Top-Level Categories
| Key | Type | Description |
|:---|:---|:---|
| `inference` | Object | LLM endpoints, active models, and brain modes. |
| `agentModes` | Object | Authoritative mode settings (RP, Hybrid, Assistant). |
| `agent` | Object | Agent profiles, system prompts, and MCP assignments. |
| `mcpServers` | Array | Registered MCP server configurations (command, env, root). |
| `storage` | Object | Database providers (Chroma, Qdrant) and local paths. |
| `firewall` | Object | Security rules for PII and API key redaction. |

### 3.2. Core Configuration Blocks
**`agentModes`**:
- `activeMode`: `'rp' | 'hybrid' | 'assistant'`
- `modes`: Specific flags for each mode (e.g., `allowRag`, `autoUseTools`).

**`mcpServers`**:
- `id`: Unique identifier.
- `command`: The command to start the server (e.g., `python`, `node`).
- `args`: Command-line arguments.
- `env`: Environment variables.

## 4. Contract Rules

### 4.1. Atomicity
Writes to `app_settings.json` are atomic. The `SettingsManager` writes to a `.tmp` file and performs a rename to prevent corruption during system crashes.

### 4.2. Immutability & Caching
-   **Cache TTL**: 30000ms (30 seconds). Full-settings reads (`loadSettings`) that fall outside this window cause a disk reload; reads within the window are served from memory.
-   **Mode reads are cache-presence-based**: `getActiveMode()` returns the in-memory value as long as any prior load has populated the cache, regardless of TTL age. Because `activeMode` can only change through `setActiveMode → saveSettings`, which always refreshes the cache, TTL-based invalidation is unnecessary for mode reads and would only generate log noise.
-   **Deep Merge**: Updates are applied via deep merging to preserve existing keys that aren't explicitly modified.

### 4.3. Authoritative State
The `getActiveMode` and `setActiveMode` methods in `SettingsManager` are the **exclusive** source of truth for the system's operational state. Direct modification of the file is discouraged while the application is running.

## 5. Security Redaction
Sensitive fields (e.g., specific API keys) are detected and redacted before logging or UI surfacing via the `firewall` pattern recognition engine.
