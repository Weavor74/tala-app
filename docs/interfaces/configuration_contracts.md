# Configuration Contracts Document — Tala System

**Document ID**: TALA-ICD-004  
**Version**: 1.0.0

## 1. Overview
This document defines the schema and mutation rules for the Tala configuration system. All persistent settings are stored in `app_settings.json` and managed by the `SettingsManager` service.

## 2. Storage Location
-   **Path**: `%USERDATA%/app_settings.json`
-   **Format**: JSON

## 3. Schema Definition (Simplified)

### 3.1. Top-Level Categories
| Key | Type | Description |
|:---|:---|:---|
| `inference` | Object | LLM endpoints, active models, and brain modes. |
| `agentModes` | Object | Authoritative mode settings (RP, Hybrid, Assistant). |
| `agent` | Object | Agent profiles, system prompts, and MCP assignments. |
| `mcpServers` | Array | Registered MCP server configurations (command, env, root). |
| `storage` | Object | Database providers (Chroma, Qdrant) and local paths. |
| `firewall` | Object | Security rules for PII and API key redaction. |

### 3.2. Core Configuration Blocks
**`agentModes`**:
- `activeMode`: `'rp' | 'hybrid' | 'assistant'`
- `modes`: Specific flags for each mode (e.g., `allowRag`, `autoUseTools`).

**`mcpServers`**:
- `id`: Unique identifier.
- `command`: The command to start the server (e.g., `python`, `node`).
- `args`: Command-line arguments.
- `env`: Environment variables.

## 4. Contract Rules

### 4.1. Atomicity
Writes to `app_settings.json` are atomic. The `SettingsManager` writes to a `.tmp` file and performs a rename to prevent corruption during system crashes.

### 4.2. Immutability & Caching
-   **Cache TTL**: 2000ms. Identical reads within this window are served from memory.
-   **Deep Merge**: Updates are applied via deep merging to preserve existing keys that aren't explicitly modified.

### 4.3. Authoritative State
The `getActiveMode` and `setActiveMode` methods in `SettingsManager` are the **exclusive** source of truth for the system's operational state. Direct modification of the file is discouraged while the application is running.

## 5. Security Redaction
Sensitive fields (e.g., specific API keys) are detected and redacted before logging or UI surfacing via the `firewall` pattern recognition engine.

## 6. User Profile Schema (`user_profile.json`)

### 6.1. Storage Location
- **Path**: `%USERDATA%/data/user_profile.json`
- **Format**: JSON
- **Managed By**: `electron/services/UserProfileService.ts`
- **Edited Via**: `src/renderer/components/` user profile UI — not modified directly.

### 6.2. Security Context
`user_profile.json` contains PII and is never sent directly to LLMs. `UserProfileService` extracts a `UserIdentityContext` (UUID + display name only) for agent prompt injection. The full `FullUserProfilePII` object is restricted to local storage and backend services.

### 6.3. Schema Definition (`FullUserProfilePII`)

| Key | Type | Description |
|:---|:---|:---|
| `userId` | `string` (UUID v4) | Stable canonical identifier. Auto-generated on first boot if absent. |
| `firstName` | `string` | User's given name. |
| `middleName` | `string` | User's middle name. |
| `lastName` | `string` | User's family name. |
| `dateOfBirth` | `string` (ISO `YYYY-MM-DD`) | Used exclusively by the Astro Engine. Redacted from all other logs. |
| `placeOfBirth` | `string` | Used by the Astro Engine for birth chart calculation. |
| `rpName` | `string` | Roleplay / character persona name. |
| `address` | `Address` | Street, unit, city, state, zip, country. |
| `email` | `string` | Contact email. |
| `phone` | `string` | Contact phone. |
| `workHistory` | `Job[]` | Array of employment records (company, role, startDate, endDate, description). |
| `schools` | `School[]` | Array of educational records (institution, degree, yearGraduated). |
| `hobbies` | `string[]` | List of hobby strings. |
| `network` | `Contact[]` | List of contact records (name, relation, contactInfo, notes). |
| `schemaVersion` | `number` | Schema version integer for migration compatibility. |
| `lastUpdated` | `string` (ISO timestamp) | Timestamp of the most recent profile save. |

### 6.4. Mutation Rules
- All writes go through `UserProfileService.save()`, which validates UUID format before persisting.
- If `userId` is absent or invalid (non-UUID), a new UUID is generated automatically.
- Reads are performed via `UserProfileService.getFullProfile()` (returns full PII) or `getUserIdentityContext()` (returns prompt-safe subset only).
- Direct file editing while the application is running may cause data loss; always use the UI.

### 6.5. Prompt-Safe Subset (`UserIdentityContext`)

The LLM-facing identity object contains only:

| Key | Type | Description |
|:---|:---|:---|
| `userId` | `string` (UUID v4) | Stable UUID for memory graph resolution. |
| `displayName` | `string` | Preferred name for agent address. |
| `aliases` | `string[]` | Alternative names (e.g., `[firstName, rpName]`). |
