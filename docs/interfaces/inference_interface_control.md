## 1. Overview
This document defines the **Inference Interface** (the "Brain") of the Tala system. It specifies how the application logic interacts with Large Language Models (LLMs), including message formats, tool calling protocols, and streaming responses.

## 2. Architecture
The inference layer is abstracted via the `IBrain` interface, allowing for interchangeable backends (Local vs. Cloud).
-   **Contract Path**: `electron/brains/IBrain.ts`
-   **Implementations**: `OllamaBrain` (Local), `CloudBrain` (External).
-   **Runtime Discovery Source**: `electron/services/inference/InferenceProviderRegistry.ts`
-   **Selection Policy**: `electron/services/inference/ProviderSelectionService.ts`
-   **Renderer Refresh Path**: `src/renderer/Settings.tsx` via preload IPC methods in `electron/preload.ts`

## 3. Data Model

### 3.1. `ChatMessage`
The core unit of conversation exchange.
```typescript
interface ChatMessage {
    role: 'system' | 'user' | 'assistant' | 'tool';
    content: string;
    tool_calls?: ToolCall[];
    tool_call_id?: string;
    name?: string; // name of the tool specifically
    images?: string[]; // base64 encoded
}
```

### 3.2. `ToolCall`
The structure used by the LLM to request action execution.
```typescript
interface ToolCall {
    id: string;
    type: 'function';
    function: {
        name: string;
        arguments: any; // JSON object
    };
}
```

## 4. Operational Methods

### 4.1. `generateResponse`
**Type**: Blocking Asynchronous  
**Inputs**: `messages: ChatMessage[]`, `systemPrompt: string`, `tools?: any[]`  
**Returns**: `BrainResponse` { content, toolCalls, metadata }  
**Usage**: Short-turn interactions or headless loops where streaming is not required.

### 4.2. `streamResponse`
**Type**: Streaming Asynchronous  
**Inputs**: `messages: ChatMessage[]`, `onChunk: (callback)`, `signal?: AbortSignal`  
**Returns**: Final `BrainResponse` upon completion.  
**Usage**: Primary chat interface for real-time user feedback.

## 5. Protocols

### 5.1. Tool Calling (OpenAI-Compatible)
Tala expects the brain to follow the standard `tool_calls` / `tool_outputs` pattern. 
1.  Assistant emits `tool_calls`.
2.  System executes tools and appends `tool` role messages.
3.  Assistant processes outputs to continue the turn.

### 5.2. Multi-Modal Interaction
The system supports image inputs for vision-capable models (e.g., LLaVA, Gemini). Images are passed as base64 strings in the `images` array of a `user` message.

### 5.3. Error Recovery
Implementations must handle:
-   **Timeout**: Governed by `ollamaTimeoutMs` in settings.
-   **JSON Parsing**: Automatic retry if the agent fails to parse tool arguments.
-   **Context Truncation**: Handled by `AgentService` sliding window logic.

## 6. Runtime Model Availability Contract

### 6.1. Preference vs Runtime Truth
- `preferredModelId` (saved settings) is treated as preference only.
- `models` on `InferenceProviderDescriptor` is treated as runtime truth from live provider probes.
- A preferred model is considered usable only when it exists in the current runtime `models` list.

### 6.2. Ollama Discovery Semantics
- Probe path: `GET /api/tags` (via `probeOllama()` in `InferenceProviderRegistry`).
- On each refresh, provider model inventory is fully replaced (`desc.models = result.models`), including empty lists.
- If Ollama is unreachable, provider state is set unavailable and `models` is set to `[]` (no stale carry-forward).

### 6.3. Selection Validation
- `ProviderSelectionService._reconcileModel()` validates requested/preferred model IDs against live `provider.models`.
- If the requested model is missing while live inventory exists, Tala logs invalid preference and selects the first live model.
- Persisted preferred model names are never merged into live availability lists.

### 6.4. IPC / UI Refresh
- `inference:refreshProviders` performs a live re-probe and returns current inventory.
- Preload exposes `inferenceRefreshProviders()` and `inferenceListProviders()`.
- Settings inference UI refresh action rewrites local `knownModels` from the latest runtime inventory and clears stale entries when providers/models disappear.
