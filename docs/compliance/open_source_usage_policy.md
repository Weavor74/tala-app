# Open Source Usage Policy — Tala System

**Document ID**: TALA-CMP-005  
**Version**: 1.0.0  
**Status**: Formal  
**Owner**: Compliance / Security  

## 1. Overview
This document outlines the formal policy for the selection, vetting, tracking, and upgrading of open-source software (OSS) within the Tala system.

## 2. OSS Vetting and Selection
1.  **Selection Criteria**: Prefer components with active maintenance (GitHub activity), strong security history, and permissive licenses (MIT, Apache-2.0, BSD).
2.  **Safety Screening**: All new components must be scanned for known vulnerabilities (e.g., via `npm audit` or `safety` for Python) before integration.
3.  **License Review**: Any library with "Strong Copyleft" (AGPL/GPL) requires a formal architectural review to ensure it is isolated and does not compromise the proprietary status of the core application.

## 3. License Tracking
1.  **Inventory**: A Software Bill of Materials (SBOM) must be maintained and updated during each major release cycle.
2.  **Attribution**: The system must include a way (via UI or documentation) to display required copyright and license attributions for third-party libraries.
3.  **Manifest Maintenance**: All dependencies must be explicitly listed in `package.json` or a `requirements.txt` file with specified or pinned versions.

## 4. Updates and Patching
1.  **Critical Vulnerabilities**: Security patches for third-party components must be applied within 48 hours of a confirmed "High" or "Critical" vulnerability alert.
2.  **Regression Testing**: Every dependency update must be followed by a full regression suite to ensure logic stability.
3.  **Patch Management**: Usage of `patch-package` for fixing upstream bugs must be documented and submitted back to the source project where feasible.

## 5. Compliance Enforcement
Failure to follow the vetting and isolation protocols for high-risk licenses is considered a critical engineering failure and will trigger a formal code review cycle.
