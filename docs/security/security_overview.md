# Security Overview — Tala System

**Document ID**: TALA-SEC-001  
**Version**: 1.0.0

## 1. Executive Summary
Tala is designed with a **Local-First Trust Model**, prioritizing data sovereignty and local inference to minimize the exposure of user data to third-party services. The system employs a multi-layered defense-in-depth architecture to protect the host machine and user secrets from both external threats and potentially unpredictable Large Language Model (LLM) behavior.

## 2. Core Security Pillars

### 2.1. Process Isolation (Renderer vs. Main)
Tala utilizes Electron's multi-process architecture to isolate the User Interface (Renderer) from the core system logic (Main Process). 
-   **Renderer**: Untrusted. Denied access to the filesystem, network, and child process execution.
-   **Main Process**: Trusted. Executes system commands and manages data persistence under strict policy control.

### 2.2. IPC Mediation
All communication between the UI and the system is mediated by a centralized **Secure IPC Bridge**. This bridge employs channel whitelisting and input validation, preventing a compromised UI from requesting unauthorized actions (e.g., arbitrary shell execution).

### 2.3. Tool Execution Boundaries
The **ToolService** enforces strict boundaries on what the AI agent can do. Legacy tools are blocked, and all file/shell operations are sandboxed through the `CodeAccessPolicy` engine, which prevents directory traversal and shell command chaining.

### 2.4. MCP Sidecar Process Model
The **Model Context Protocol (MCP)** allows specialized logic (RAG, Astro, Mem0) to run in isolated sidecar processes. These processes communicate via `stdio` JSON-RPC, ensuring that a crash or compromise in a specialized server does not affect the main application kernel.

### 2.5. Local Inference vs. External Inference
-   **Local Inference**: Powered by Ollama or llama.cpp. Data never leaves the local machine.
-   **External Inference**: Supports Cloud providers (Gemini, Anthropic). When used, the **Quantum Firewall** automatically redacts PII and secrets before transmission.

### 2.6. Storage and Persistence Security
All system configurations and agent data are stored locally in the `%USERDATA%` directory. Writes are atomic to prevent corruption, and sensitive data (like chat history) is protected by host-level filesystem permissions.

## 3. High-Level Security Flow
1.  **Input**: User input is validated for prompt injection via Guardrails.
2.  **Orchestration**: AgentService coordinates tools under strict `CodeAccessPolicy`.
3.  **Redaction**: The Quantum Firewall (log_redact) scrubs secrets before any logging or outbound transmission.
4.  **Audit**: Every inference and tool call is recorded in a tamper-resistant local audit log.
