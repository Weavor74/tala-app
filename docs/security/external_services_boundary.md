## 1. Overview
Tala is a **Local-First** application. This document defines the exceptions to this rule by detailing all outbound network communication and external process dependencies.

## 2. External Interaction Matrix

| Service | Purpose | Data Sent | Data Received | Security Implication |
|:---|:---|:---|:---|:---|
| **Ollama RPC** | Local Inference | Prompts, tool specs. | Text tokens. | Minimum if bound to `127.0.0.1`. |
| **Gemini / Anthropic API** | Cloud Inference | Scrubbed prompt, context. | Model responses. | Data leaves local trust boundary. |

| **Update Check** | Maintenance | Component versions. | "New Version" flag. | Disclosure of app version. |

## 3. Operational Air-Gap Capabilities

### 3.1. Offline Mode
Tala can operate in a **Fully Offline** state if:
1.  An Ollama/Llama.cpp local engine is configured.
2.  Local RAG (tala-core) and Memory (mem0-core) servers are used.
3.  Cloud API keys are removed from settings.
In this mode, zero network traffic is generated beyond local loopback (`127.0.0.1`).

### 3.2. Cloud-Transit Protection
When used with Cloud providers, the **Quantum Firewall** enforces the following:
- **Redaction**: PII and Secrets are scrubbed from the `ChatMessage` content before the API request is constructed.
- **TLS**: Mandatory HTTPS (TLS 1.2+) is enforced by the Node.js `https` module.
- **Payload Minimization**: Only the minimum required context is sent to the provider.

## 4. MCP Sidecar Trust
MCP servers are spawned as local child processes. 
- **Constraint**: MCP servers are denied network sockets by default in the system prompt instructions.
- **Risks**: A malicious or un-audited MCP server could attempt to initiate its own network connections. Verification of MCP server binaries is a recommended hardening step.
