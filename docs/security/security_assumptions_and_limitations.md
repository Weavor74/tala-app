# Security Assumptions and Limitations — Tala System

**Document ID**: TALA-SEC-010  
**Version**: 1.0.0  
**Status**: Formal  
**Owner**: Architecture  

## 1. Security Assumptions
The security posture of Tala is built upon the following foundational assumptions about the environment and user behavior:

1.  **Workstation Trust**: It is assumed that the user's host machine is physically secure and not already compromised by a rootkit, keylogger, or remote access trojan (RAT).
2.  **OS Kernel Integrity**: We assume the underlying Operating System (Windows/Mac/Linux) is functioning correctly and its security features (ASLR, DEP, DAC) are not bypassed.
3.  **Local Administrator Trust**: The system assumes that any user with administrative access to the host machine has the power to bypass Tala's local security controls (e.g., direct editing of `app_settings.json`).
4.  **Provider Compliance**: When using Cloud Brains, it is assumed that the model providers (Google, Anthropic) comply with their documented privacy policies.
5.  **User Responsibility**: It is assumed that users do not intentionally disable security features or hardcode secrets into public repositories handled by the agent.

## 2. Documented Limitations

### 2.1. Protection Against Host Compromise
Tala is a user-mode application. It **cannot** protect data if the host OS is compromised. An attacker with system-level access can read process memory, intercept IPC traffic, and modify agent history.

### 2.2. Perfect Redaction vs. Novel PII
The **Quantum Firewall** is a rule-based redaction engine. While it is effective against common secrets (API keys, Tokens), it may fail to redact novel PII formats or data hidden within complex unstructured text.

### 2.3. Log Immutability
Currently, logs are stored as plain JSON on disk. There is no cryptographical guarantee of log immutability; an attacker with write access can edit the audit trail to conceal their presence.

### 2.4. Agent Mode Sovereignty
While "Hybrid" and "RP" modes are more secure, the transition between modes is governed by the `SettingsManager`. If the settings file is tampered with, the agent's behavior may deviate from the user's intended mode.

### 2.5. Reflection Subsystem Hardening
The Reflection and Goal Graph subsystems are currently in a "Development" state. Their security controls are less mature than the core `AgentService` and `ToolService` layers.
